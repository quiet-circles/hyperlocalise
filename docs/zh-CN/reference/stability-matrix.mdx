使用此页面来确定哪些功能适合作为生产环境的默认设置，哪些功能应继续置于更严格的发布控制之下。

## 功能成熟度

| 区域 | 范围 | 状态 | 备注 |
| --- | --- | --- | --- |
| LLM 提供商层 | OpenAI、Azure OpenAI、Gemini、Anthropic、AWS Bedrock、LM Studio、Groq、Ollama | 稳定 | 主要生成路径用于 `run`. |
| 计划-第一 + 锁文件 | `run --dry-run`，确定性规划， `.hyperlocalise.lock.json` 检查点 | 稳定 | CI 和 PR 审查流程的推荐默认设置。 |
| TMS 适配器 | Crowdin、LILT AI、Lokalise、Phrase、POEditor、Smartling | 实验性 | `sync push` / `sync pull` 合约可能会演变。在 CI 中固定 CLI 版本。 |
| 评估框架 | `eval run`, `eval compare`，质量/回归评分 | 实验性 | 报告架构和评分可能会在不同版本之间发生变化。 |
| CI-就绪状态标签 | `ready`, `needs review`, `missing` 源自 `status` CSV | 实验性 | 当前文档的标签映射是稳定的，但分类语义可能会演变。 |

## 推荐的发布策略

1. Treat `run` + 在默认的 CI 检查中将 lockfile 作为基线。
2. 使用 `sync` 和 `eval` 在受控的流水线中，并明确固定版本号。
3. 通过状态门控发布-启用写入前的基础检查-有能力 `sync` 作业。
