# Storage System (Storage Adapters + Sync Model)

This package defines the shared translation storage model used by `hyperlocalise` to sync with remote systems (TMS/custom backends) and local files.

It is the contract between:

- local translation storage (JSON files + metadata sidecars)
- remote storage adapters (for example `poeditor`)
- sync orchestration (`sync pull` / `sync push`)
- LLM translation generation workflows

## Why this exists

`hyperlocalise` has two different jobs:

1. Generate translations (LLM)
2. Sync curated translations with external systems (TMS)

Those jobs should not depend on a provider-specific schema.

The storage system solves this by using a normalized entry model and a `StorageAdapter` interface.

## Core concepts

### Normalized entry

A translation is represented as a normalized entry:

- `Key` (translation key / term)
- `Context` (optional disambiguation)
- `Locale`
- `Value`
- `Provenance` (who produced it and state: draft/curated)
- `Remote` (optional remote adapter metadata)

Identity is:

- `(key, context, locale)`

This identity is used for diffing, conflict detection, and metadata sidecars.

### StorageAdapter

Remote providers implement a small interface:

- `Pull(...)` download remote translations into normalized entries
- `Push(...)` upload normalized entries to the remote provider
- `Name()`
- `Capabilities()`

Adapters should only map provider API <-> normalized entries.
Conflict logic belongs in `syncsvc`.

### Local storage and metadata sidecars

Local translation values remain in normal files (for example `lang/fr.json`).

Provenance and curation state are stored in sidecar files:

- `lang/fr.json` (values)
- `lang/fr.meta.json` (metadata)

This avoids polluting translation values with operational metadata.

## Data model (relevant fields)

### `EntryProvenance`

Tracks translation origin and status:

- `origin`: `llm`, `human`, `imported`, `unknown`
- `state`: `draft`, `curated`
- `updated_at`
- `updated_by`

### `RemoteMeta`

Tracks remote metadata when available:

- `adapter` (for example `poeditor`)
- `revision`
- `updated_at`

## LLM Translation Flow (Recommended)

This is the intended workflow when using LLM translation + TMS curation.

### Current implementation status

The current implementation supports the sync side of this flow (`sync pull` / `sync push`) and local sidecar metadata storage.

Important current limitation:

- the LLM translation pipeline is not yet wired to automatically write provenance metadata sidecars (`origin=llm`, `state=draft`)

This means:

- the flow works best when metadata already exists, or when you add metadata as part of your LLM step
- local entries without provenance are treated conservatively during conflict checks

### 1. LLM generates local drafts

Your translation pipeline writes local values (for example `lang/fr.json`) and metadata sidecars:

- `origin=llm`
- `state=draft`

Example metadata intent:

- key `checkout.submit`, locale `fr`
- value generated by LLM
- marked as draft until reviewed

### 2. Push drafts to TMS (optional)

`sync push` sends local normalized entries to the remote storage adapter.

v1 behavior is conservative:

- local drafts do not overwrite curated remote values silently
- conflicts are reported/skipped

### 3. Human reviews/curates in TMS

A translator or reviewer updates translations in the remote system (for example POEditor).

These are considered curated/human edits.

### 4. Pull latest curation

`sync pull` downloads remote curated entries and merges them into local storage.

Default intent:

- curated remote values can replace local LLM drafts
- local curated mismatches are conflicts (report-only)
- missing provenance on local mismatches is treated conservatively (conflict)

When applied, local metadata is updated to:

- `origin=human`
- `state=curated`

## Example flow (high level)

1. Generate drafts with LLM into `lang/fr.json`
2. Save metadata in `lang/fr.meta.json` (`origin=llm`, `state=draft`)
3. Run `hyperlocalise sync push --dry-run`
4. Review conflicts and push safe entries
5. Curate translations in TMS
6. Run `hyperlocalise sync pull --dry-run`
7. Apply pull to update local values + metadata sidecars

## ASCII flow (generic TMS adapter flow)

```text
                         +--------------------------+
                         |       source JSON        |
                         |      (e.g. en.json)      |
                         +------------+-------------+
                                      |
                                      | LLM translate (outside sync)
                                      v
                    +-----------------------------------+
                    | local target JSON (e.g. fr.json)  |
                    | + sidecar meta (fr.meta.json)     |
                    | origin=llm, state=draft           |
                    +------------+----------------------+
                                 |
                 sync push       |       sync pull
              (local -> remote)  |   (remote -> local)
                                 |
                +----------------+----------------+
                v                                 v
      +----------------------+          +----------------------+
      |   syncsvc diff +     |          |   syncsvc diff +     |
      | conflict decisions   |          | conflict decisions   |
      +----------+-----------+          +----------+-----------+
                 |                                 |
                 | uses StorageAdapter             | uses StorageAdapter
                 v                                 v
      +----------------------+          +----------------------+
      |   TMS adapter        |<-------->|    Remote TMS        |
      | (POEditor,           |          | (POEditor,           |
      |  Smartling, etc.)    |          |  Smartling, etc.)    |
      | map provider <->     |          | human curation       |
      | normalized entries   |          | reviewed strings     |
      +----------------------+          +----------------------+
```

Conflict decisions (current, summarized):

- Pull:
  - local `llm/draft` + remote curated mismatch -> update local (on apply)
  - local curated mismatch -> conflict
  - missing provenance + mismatch -> conflict
- Push:
  - local draft vs remote curated mismatch -> conflict by default (`--force-conflicts` can override)
  - remote missing key -> create candidate
  - same value -> unchanged
  - safe mismatches -> update candidate
  - invariant violations -> conflict (always blocked)

## How the flow works with the current implementation

`hyperlocalise` currently assumes the local JSON files are the starting point of truth for file content, and sync operations compare local files against the remote storage adapter snapshot.

### A. Local source JSON exists

Example:

- `lang/en.json` source (reference source locale)
- `lang/fr.json` target locale file

The sync implementation reads target locale JSON files (for example `lang/fr.json`) and a metadata sidecar if present:

- `lang/fr.meta.json`

### B. LLM translation step runs (outside sync)

The current sync code does not invoke the LLM translation process.

Expected integration pattern:

1. Read source JSON (for example `lang/en.json`)
2. Generate target translations with LLM
3. Write target file (for example `lang/fr.json`)
4. Write/update sidecar metadata with:
   - `origin=llm`
   - `state=draft`

If step 4 is skipped, sync still works but conflict handling becomes more conservative.

### C. `sync push` compares local vs remote before upload

`sync push` flow (current implementation):

1. Read local JSON values + sidecar metadata (`localstore.BuildPushSnapshot`)
2. Pull remote baseline from storage adapter (`StorageAdapter.Pull`)
3. Diff by `(key, context, locale)`
4. Build push plan
5. Skip/report conflicts
6. Push only safe entries via `StorageAdapter.Push` (unless `--dry-run`)

### D. Human curation happens in storage (POEditor)

Remote translators/reviewers update terms in POEditor.

The POEditor adapter returns pulled entries as curated/human by default:

- `origin=human`
- `state=curated`

### E. `sync pull` updates local values with latest curation

`sync pull` flow (current implementation):

1. Read local JSON values + sidecar metadata (`localstore.ReadSnapshot`)
2. Pull remote curated entries from POEditor (`StorageAdapter.Pull`)
3. Diff by `(key, context, locale)`
4. Apply conservative curation-aware decision rules
5. Report conflicts (and fail if `--fail-on-conflict`)
6. If not dry-run, write:
   - updated target JSON values
   - updated metadata sidecar entries

## Sync behavior (v1)

- conflict-first, no silent overwrite by default
- `pull` and `push` are separate operations (no full bidirectional reconcile command yet)
- remote adapter behavior is provider-specific, but merge/conflict rules are centralized

## Conflict decision rules (current implementation)

The sync service uses provenance-aware conflict decisions in `syncsvc`.

### `sync pull` (remote -> local)

Decision key:

- local value
- remote value
- local provenance (`origin`, `state`)

Current rules:

1. Local key missing, remote exists
   - result: `create`
   - action: add local value and metadata (on apply)

2. Local and remote value equal
   - result: `unchanged`

3. Local is `origin=llm`, `state=draft`, remote value differs
   - result: `update` (remote curated can replace local draft)
   - action on apply:
     - write remote value
     - set metadata to `origin=human`, `state=curated`

4. Local is `state=curated`, remote value differs
   - result: `conflict`
   - reason: curated mismatch must be reviewed manually

5. Local provenance is missing/unknown and value differs
   - result: `conflict`
   - reason: conservative behavior when origin/state is unknown

6. Local provenance is non-LLM and non-curated (for example imported/manual), value differs
   - current behavior: treated as updatable by remote curated value
   - note: this may be tightened later if you want stricter manual review

### `sync push` (local -> remote)

Current rules:

1. Remote key missing, local exists
   - result: `create` candidate (safe to push)

2. Local and remote value equal
   - result: `unchanged`

3. Local is `draft` and remote is `curated`, values differ
   - default: `conflict`
   - with `--force-conflicts`: `update` candidate

4. Local provenance is missing/unknown, values differ
   - default: `conflict`
   - with `--force-conflicts`: `update` candidate

5. Other mismatches
   - result: `update` candidate

6. Any invariant violation
   - result: `conflict` (always blocked)

### Operational flags affecting behavior

- `--dry-run` (default `true`)
  - compute/report only, no local writes or remote pushes
- `--fail-on-conflict` (default `true`)
  - return error when conflicts are found
- `--apply-curated-over-draft` (default `true` for pull command)
  - controls whether remote curated values can replace local drafts during pull
- `--force-conflicts` (default `false` for push command)
  - allows overwrite for mismatch conflicts in push
  - does not bypass invariant validation safeguards

## Conflict examples (decision table)

### Pull examples

| Local provenance | Local value | Remote value | Result |
| --- | --- | --- | --- |
| `llm/draft` | `Bonjour` | `Salut` | `update` (remote curated replaces draft) |
| `human/curated` | `Bonjour` | `Salut` | `conflict` |
| `unknown` | `Bonjour` | `Salut` | `conflict` |
| any | `Bonjour` | `Bonjour` | `unchanged` |

### Push examples

| Local provenance | Local value | Remote provenance | Remote value | Result |
| --- | --- | --- | --- | --- |
| `llm/draft` | `Salut` | `human/curated` | `Bonjour` | `conflict` (or `update` with `--force-conflicts`) |
| any | `Bonjour` | any | missing | `create` candidate |
| any | `Bonjour` | any | `Bonjour` | `unchanged` |
| non-draft mismatch | `Salut` | non-curated or unknown | `Bonjour` | `update` candidate |

## Package layout (related)

- `internal/i18n/storage`: normalized model + `StorageAdapter` interface
- `internal/i18n/storageregistry`: adapter registry/factory
- `internal/i18n/syncsvc`: diff/conflict/merge orchestration
- `internal/i18n/localstore`: local JSON + sidecar metadata store
- `internal/i18n/storage/poeditor`: POEditor adapter
- `internal/i18n/storage/lokalise`: Lokalise adapter
- `internal/i18n/storage/smartling`: Smartling adapter
- `internal/i18n/lockfile`: optional sync checkpoints (not source of truth)
